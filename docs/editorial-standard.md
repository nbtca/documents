## 为什么有这份标准

AI 繁荣的当下，愿意手写文档的人越来越少——这没关系。但 AI 不知道所有约束，会随处乱写；真人没有规范也一样。没有成文约束，站点会退回到「看似专业、其实脑补」的内容。而 NBTCA 是一个把细节与人文当回事的社区，垃圾内容会让人离开。

这份标准让文档的**严谨与可读可被检查、可被守住**。我们不是 Wikipedia 那样的大组织，但我们借鉴它的经验：可供查证（Verifiability）、不搞原创研究（No Original Research）、中立（NPOV）。

## 北极星

一句话：**To be at the intersection of technology and liberal arts.**

读者的感受不靠煽情制造，而由细节自然长出。四条价值，每条都要能在每一页上被检查：

| 价值         | 落到写作上                                   |
| ------------ | -------------------------------------------- |
| 我们的行文   | 清晰、不啰嗦、面向完全不懂的陌生人           |
| 对真相的严谨 | 可查证、反脑补                               |
| 对技术的热爱 | 讲透原理、用对工具、不糊弄                   |
| 人文关怀     | 隐私最小化、求助有边界、中立不劝诱、优雅降级 |

## 支柱一 · 写作纪律

- **一篇一职。** 每篇只回答一个层级的问题。主流程文只讲判断与导航（我该不该做、按什么顺序、去哪找）；可勾选、可打印的执行件放进材料卡；一句话名词解释放进概念词条。
- **首段即答案。** 开篇第一段就是自足的结论或定义，读完首段即懂大概。概念词条尤其严格：首段那句定义会被悬停卡自动取用——写不好，卡片当场露馅。
- **面向新人。** 不预设内幕知识。黑话首次出现即链到概念词条，或给一句解释。
- **清单只活一处。** 同一份 checklist、分工表只存在于一个地方；别处只留一句摘要加链接。重复迟早打架。
- **中立陈述。** 有争议的事（例如活动与第二课堂学分、志愿者时长挂钩）只客观记录现状，不替任何一派站队、不劝诱。
- **主动语态、平实动词、句子式大小写。** 一个元素只做一件事：标签只标注，示例只演示。

## 支柱二 · 事实验证

- **只写做过或确认过的（反脑补铁律）。** 流程必须从 NBTCA 真实运作里长出来。答不上「这条你问过谁、在哪见过」的，就不许写成事实。凭空发明流程形状，本身即不合格。
- **查不到就标注，不编造。** 缺口留成「待核实」并指向具体的人或官方来源，绝不用一个自信的假句子或空占位符糊弄。缺口摆明面上，是邀请人来补。
- **分清长期判断与当年事实。** 「涉及审批就早准备」是长期判断，稳定、少引用；「某系统入口、某审批人」是当年事实，易变。
- **易变事实集中一处，带核对日期。** 当年会变的事实收进对应概念词条，附官方来源与 `最后核对：YYYY-MM`。过期只烂在一处，不污染主流程文的判断骨架。
- **易变的人、岗位、联系人也抽成词条。** 不在正文里散落「当年为 X」；「谁是对接老师」「谁负责报销」这类随人事变动的信息，收进对应词条、正文只链接过去——人一换，只改那一个词条。
- **溯源靠人，不靠计时器。** 文档的变更是事件驱动（机构、人员变动），不是时间驱动。每页保留真实「维护信息」（谁负责、该问谁），不留 `???`。
- **修订历史交给 git。** 「谁改了什么」由 git blame、GitHub 历史与 VitePress `lastUpdated` 自动记录。维护信息块是 stewardship（对这页负责的人），不是逐行 changelog，不要手写。

## 结构与维护约定

- **顶层信息架构**：认识 NBTCA · 上手 · 教程 · 流程 · 维修 · 参考 · 存档。
- **概念词条**住在 `concepts/`（导航「参考」），**不放 `archived/`**。`archived/` 只留真历史；仍在生效的手册、身份文档应迁回「教程」或「认识 NBTCA」。
- **概念词条格式**：`# 标题` → `维护信息` → 首段定义（自足，供悬停卡）→ 相关来源 →（若易变）`最后核对` 状态。
- **不做大规模迁移。** 新内容 100% 守标准；旧内容只在你本来就要动它时顺手升级（童子军原则）。缺口用红链或存根标出，不假装完整。
- **历史文档保原貌，活跃文档要审阅。** 会议纪要、往届计划、旧纲要等历史材料不做大规模修正；活跃与技术文档应持续审阅、更新行文——简洁、准确、剔除 AI 生成的冗余与套话。整理这个站，就是在建社区自己的知识库。
- **小而集中的 PR。** 见 [CONTRIBUTING](../CONTRIBUTING.md)：内容、导航、CI、资产尽量分开提交，便于审阅与回滚。
- **维护信息默认链接 GitHub 主页**（隐私）。不在文档里硬编私人邮箱或 QQ；愿意公开邮箱的，本人可放在自己的 GitHub 主页。
- **站内链接即词条（wiki 模型）**。链接只有两档：**站内**（都是可悬停预览的「词条 / 文章」，虚线样式）与**站外**（尾随 ↗）。因此每个站内可链接的页面都要有一句**自足的首段**（供预览取用），标题也要能独立读懂——不合适就调整。

## 合并前自检（AI 和真人都先过一遍）

- [ ] 这篇只做一件事吗？清单或表格有没有和别处重复？
- [ ] 首段读完能懂大概吗？（概念词条：首段是不是一句自足定义？）
- [ ] 新人会不会被没解释的黑话挡住？**站内术语在首次出现处链了概念、站外工具/服务/标准给了外链**吗？（只在首次出现处链，避免过度链接）
- [ ] 每条时效性或官方断言，答得上「问过谁、哪来的」吗？查不到的是否标注而非编造？
- [ ] 易变事实是否收进词条、带来源和 `最后核对` 日期？
- [ ] 有争议处是否中立、无劝诱？是否只按必要范围收集与公开个人信息？
- [ ] 这页有真实维护信息（不是 `???`）吗？
- [ ] 站内链接、sidebar，以及 `pnpm docs:build` 和 `pnpm run ci:lint` 都通过吗？
